Project 01

JARVIS

A local-first AI assistant built to automate real computer systems.

Under active development

Deterministic routing allows simple system commands to execute almost instantly without requiring language model inference.

Why I Built It

Most AI assistants try to solve every problem by sending everything through a language model. Opening a browser tab or launching an application shouldn't take seconds of inference time.

JARVIS was built around a simple idea: use AI only when AI is actually needed.

The Problem

Early versions were slow, unpredictable and occasionally read their own reasoning process aloud through the text-to-speech pipeline.

That forced me to redesign how requests are routed, separating simple system actions from complex reasoning tasks.

What It Taught Me

Building JARVIS taught me that AI products are often software engineering problems before they're machine learning problems.

Architecture decisions, latency, threading and good UX matter just as much as model selection.

How Requests Are Routed

Python PyQt6 Whisper SQLite Playwright Ollama Piper TTS Win32 API

Currently Building: Improving browser automation, multi-step task execution, and local AI capabilities.

Project 02

DeathLeade Network

Automated Game Server Commerce Platform

Frontend deployed · Backend currently offline

Visit Website →

The platform was designed to automate package delivery through TCP RCON once payment verification is completed.

What I Built

DeathLeade Network is a Minecraft gaming network storefront that was designed to automate the complete purchasing experience. Players can purchase in-game ranks and packages through the website and receive their rewards automatically once the backend services are deployed.

The backend was built to validate purchases, store order data, communicate with the Minecraft server, and automate package delivery through TCP RCON.

The Challenge

Building a game commerce platform is more than building a website. Payment processing, database consistency, real-time server communication, and automated delivery systems must work together reliably for every order.

Hosting limitations prevented direct TCP communication with the game server, forcing me to redesign the backend architecture and separate frontend and server-side responsibilities.

What I Learned

This project introduced me to production-level problems that most student projects never encounter. I learned how hosting providers affect application architecture, how backend services should be designed independently from the frontend, and how systems behave when real users are interacting with them simultaneously.

I also gained hands-on experience working with TCP protocols, deployment strategies, database optimization, and building reliable automation pipelines for real-world use cases.

Architecture Overview

Node.js Express MongoDB TCP RCON Railway REST APIs Discord Webhooks

Status: Frontend deployed. Backend deployment is currently paused due to hosting costs.
